# FinSpark 评估体系产品经理指南：Langfuse vs OpenEvals vs Promptfoo

> **版本**: v1.0  
> **日期**: 2026-04-07  
> **读者**: 产品经理 / 非技术决策者  
> **目的**: 用产品思维理解三个工具的角色、边界和价值

---

## 一句话总结

> **Langfuse 看得见系统在干什么，OpenEvals 判得了质量好不好，Promptfoo 守得住版本不退步。**

如果把 FinSpark 比作一个餐厅：

| 工具 | 餐厅类比 | 一句话 |
|------|---------|--------|
| **Langfuse** | 后厨监控摄像头 | 实时看到每道菜在哪个工位、花了多久、用了多少原料 |
| **OpenEvals** | 试菜评委 | 随机抽几道菜，按口味/摆盘/用料 6 个维度打分 |
| **Promptfoo** | 新菜品上架前的品控测试 | 换了食材供应商？先用固定菜单测一轮，不达标不许上菜 |

---

## 一、三个工具解决的是三个完全不同的问题

### Langfuse — "系统跑得怎么样？" (可观测性)

**类比**: 就像你打开美团商家后台，看实时订单状态、出餐时间、原料消耗。

**它回答的核心问题**:
- 贵州茅台分析跑了多久？卡在哪个 Agent？
- 这次分析花了多少 token、多少钱？
- 上周跑的 50 次分析，哪些失败了？失败原因是什么？
- PROFITABILITY Agent 平均延迟是 3 秒还是 15 秒？

**产品经理关心的数据**:
| 你关心的问题 | Langfuse 给你的答案 |
|------------|-------------------|
| 单次分析成本 | "这次分析用了 180K tokens，花了 $1.35" |
| 系统稳定性 | "过去 7 天成功率 94%，失败主要集中在 CASH_FLOW Agent" |
| 性能瓶颈 | "FORECAST Agent P90 延迟 12 秒，是最慢的环节" |
| 模型账单 | "本月 GPT-4.1 调用 $180，GPT-4.1-mini 调用 $45" |

**什么时候用**: **每天、实时**。就像你每天看业务仪表板一样。

**FinSpark 中的实际实现**: 每次运行 12-Agent 分析时，Langfuse 自动记录:
```
Trace (贵州茅台分析)
  ├─ Span: Phase 1 (三表分析)
  │   ├─ Generation: PROFITABILITY Agent  → 3.2s, 15K tokens, $0.12
  │   ├─ Generation: BALANCE_SHEET Agent  → 4.1s, 18K tokens, $0.14
  │   └─ Generation: CASH_FLOW Agent     → 2.8s, 12K tokens, $0.10
  ├─ Span: Phase 2 (深度分析)
  │   ├─ Generation: RISK Agent          → 5.5s, 20K tokens, $0.16
  │   └─ ...
  └─ Score: reportScore = 0.82
```

---

### OpenEvals — "输出质量达标吗？" (质量评估)

**类比**: 就像米其林评审员，每道菜按固定标准打分——不是看后厨效率，而是看最终端上来的菜好不好吃。

**它回答的核心问题**:
- PROFITABILITY Agent 输出的营收分析**准确吗**？
- RISK Agent 有没有**编造**不存在的风险数据？
- FORECAST Agent 的预测**分析够深入吗**？
- 12 个 Agent 之间的结论**逻辑自洽吗**？

**产品经理关心的数据**:
| 你关心的问题 | OpenEvals 给你的答案 |
|------------|---------------------|
| 分析质量趋势 | "本周平均质量分 0.78，比上周提升 0.05" |
| 最弱环节 | "EARNINGS_QUALITY Agent 数据准确性最低，只有 0.62" |
| 幻觉率 | "5% 的分析中发现了数据编造现象" |
| 低分告警 | "报告 #234 的 RISK Agent 评分 0.31，已触发告警" |

**什么时候用**: **生产环境中实时/抽样运行**。每次生成分析报告时，有 10% 概率被评审。

**FinSpark 中的实际实现**: Agent 输出后，OpenEvals 用另一个 LLM (gpt-4.1) 做"裁判":
```
PROFITABILITY Agent 输出 → 送给 6 个"评委" 打分:
  ├─ 数据准确性 (25%): 0.85 — "营收数据与年报一致"
  ├─ 分析深度   (25%): 0.72 — "缺少同比趋势分析"
  ├─ 专业洞察   (20%): 0.78 — "毛利率分析到位"
  ├─ 逻辑一致性 (15%): 0.90 — "数据间无矛盾"
  ├─ 表达质量   (10%): 0.88 — "结构清晰"
  └─ 幻觉检测    (5%): 0.95 — "未发现编造数据"
  
  加权总分: 0.82
  → 记录到 D1 数据库
  → 同步到 Langfuse Score
  → 低于 0.5 触发告警
```

---

### Promptfoo — "改了代码会不会变差？" (回归防护)

**类比**: 就像 iPhone 出厂前的质检流程——每部手机都要跑同一套测试，通过了才能出货。不是看手机好不好用（那是用户评价），而是确保"不比上一版差"。

**它回答的核心问题**:
- 我改了 RAG 检索代码，搜索结果会不会变少？
- 新版本的回答质量有没有比旧版本差？
- 如果换一种 Rerank 策略，factuality 会提高还是下降？
- 哪个配置 (topK=8 vs topK=12) 效果更好？

**产品经理关心的数据**:
| 你关心的问题 | Promptfoo 给你的答案 |
|------------|---------------------|
| 版本对比 | "v10.3 比 v10.2 factuality 提升 12%，sourceCount 从 4 升到 8" |
| 代码安全 | "这次 PR 改了检索逻辑，23 题全部通过，可以合并" |
| 配置选型 | "Rerank ON 的 faithfulness 比 OFF 高 0.15，建议开启" |
| 回归检测 | "CI 拦截: source count 降到 4，不允许合并代码" |

**什么时候用**: **开发阶段，每次提交代码时**。在代码合并到主分支之前自动运行。

**FinSpark 中的计划实现** (尚未开发):
```
开发者修改 ragPipeline.ts → 提交 PR → GitHub Actions 自动触发:

  Promptfoo 用固定的 23 道题测试:
  ├─ Q1: "比亚迪2024年营收?" → 检查 factuality >= 0.7 ✅
  ├─ Q2: "招商银行净利润?" → 检查 factuality >= 0.7 ✅
  ├─ Q3: "宁德时代季度收入?" → 检查 sourceCount >= 6 ❌ (只有 4 个!)
  └─ ...

  结果: ❌ 失败! 自动阻断 PR 合并
  → PR 评论: "RAG 回归检测失败: source count 不足"
```

## 二、三者关系：不是竞争，是流水线

```
时间线维度:

  开发阶段                              生产阶段
  (写代码时)                            (用户在用时)
  ┌────────────┐                       ┌──────────────┐   ┌──────────────┐
  │  Promptfoo  │  ──代码合并──→       │   Langfuse    │ + │  OpenEvals   │
  │  "守门员"   │                       │   "监控屏"    │   │  "评委"      │
  │             │                       │              │   │              │
  │ 跑固定题库  │                       │ 记录每次运行  │   │ 随机抽样打分  │
  │ 对比新旧版本│                       │ Token/成本    │   │ 6维度评分    │
  │ 阻断劣质代码│                       │ 延迟/错误     │   │ 幻觉检测     │
  └────────────┘                       └──────────────┘   └──────────────┘

  问: 该不该合入代码?                   问: 系统健康吗?    问: 质量达标吗?
```

**更直白的比喻**:

| 阶段 | 对应工具 | 生活类比 |
|------|---------|---------|
| 面试时 | **Promptfoo** | 候选人做一套笔试题，不及格不发 offer |
| 在职时 | **Langfuse** | 员工的考勤打卡、工时记录、差旅报销 |
| 季度考核 | **OpenEvals** | 随机抽取员工的工作成果，按 KPI 打分 |

**三者完全不冲突，因为它们工作在不同时间点、回答不同问题**。

## 三、从产品经理视角看价值

### 3.1 如果只能选一个？

| 优先级 | 选择 | 理由 |
|--------|------|------|
| **必须有** | Langfuse | 没有可观测性 = 瞎子开车，出了问题完全无法排查 |
| **强烈建议** | OpenEvals | 没有质量评估 = 不知道产品好不好，只能靠用户投诉 |
| **锦上添花** | Promptfoo | 没有回归防护 = 每次上线都在赌运气 |

### 3.2 每个工具的"杀手级场景"

#### Langfuse 的杀手级场景: "贵州茅台分析失败了，为什么？"

没有 Langfuse:
> "分析失败了。" → 查日志 → 日志太多找不到 → 可能是 CASH_FLOW Agent 超时？→ 不确定 → 盲猜修复 → 验证 → 不对 → 再猜...

有了 Langfuse:
> 打开 Langfuse → 搜索 "600519.SH" → 找到那次 Trace → 看到 RISK Agent 返回了 Error → 点进去看到 "Token limit exceeded, prompt 长度 32K" → 根因明确 → 5 分钟修复

#### OpenEvals 的杀手级场景: "分析报告质量在悄悄下滑"

没有 OpenEvals:
> 用户反馈 "最近分析不太准" → 人工抽查 5 份报告 → 看起来还行 → 忽略 → 一个月后大量投诉 → 才发现是模型更新导致幻觉增多

有了 OpenEvals:
> 每天自动评估 → 趋势图显示过去一周幻觉评分从 0.95 降到 0.72 → 自动触发低分告警 → 第二天就发现问题 → 排查发现是 API 供应商偷偷切了模型

#### Promptfoo 的杀手级场景: "改了一行代码，搜索结果从 8 个变成 4 个"

没有 Promptfoo:
> 开发者修复了一个 bug → 代码审查看起来 OK → 合并上线 → 三天后评测发现 source count 下降 → 排查发现是 rerank 函数只返回了 top-4 → 已经影响了线上用户三天

有了 Promptfoo:
> 开发者提交 PR → CI 自动跑 6 道回归题 → 3 分钟后: "❌ source count = 4, 预期 >= 6" → PR 被自动阻断 → 问题在上线前就被拦截 → 用户零影响

## 四、数据维度对比

### 4.1 三者"看"的数据完全不同

| 维度 | Langfuse | OpenEvals | Promptfoo |
|------|---------|-----------|-----------|
| **运行时间/延迟** | ✅ 每个 Agent 精确到 ms | ❌ | ❌ |
| **Token 消耗/成本** | ✅ 自动计算 | ❌ | 仅记录 grader 用量 |
| **请求链路 (Trace)** | ✅ 全链路 DAG 可视化 | ❌ | ❌ |
| **输出内容质量** | ❌ (只存不判) | ✅ 6 维度打分 | ✅ 多种 assertion |
| **幻觉检测** | ❌ | ✅ 实时检测 | ✅ faithfulness 断言 |
| **版本 A/B 对比** | ❌ | ❌ | ✅ 矩阵对比视图 |
| **CI/CD 自动阻断** | ❌ | ❌ | ✅ pass/fail 门禁 |
| **配置参数对比** | ❌ | ❌ | ✅ 多 provider 矩阵 |
| **安全测试 (红队)** | ❌ | ❌ | ✅ 40+ 攻击插件 |
| **错误/异常追踪** | ✅ Error 级别标记 | ❌ | ❌ |
| **低分告警** | ❌ | ✅ 自动告警 | ❌ (CI 失败就是告警) |

### 4.2 三者的输入和输出

| 工具 | 输入 | 输出 | 存储 |
|------|------|------|------|
| **Langfuse** | Agent 执行时自动埋点 (Trace/Span/Generation) | 可视化仪表板、成本报表、延迟趋势 | Langfuse Cloud / 自建 PostgreSQL |
| **OpenEvals** | Agent 的最终输出文本 | 6 维度评分 (0-1)、告警、趋势 | D1 数据库 + Langfuse Score |
| **Promptfoo** | 固定测试题库 (YAML) | pass/fail 结果、矩阵对比、HTML 报告 | 本地 JSON + CI Artifacts |

## 五、成本对比

| 工具 | 类型 | 月成本 | 付费对象 |
|------|------|--------|---------|
| **Langfuse** | 可观测性 | $0-30 | 自部署服务器 (或免费 Cloud 额度) |
| **OpenEvals** | 质量评估 | $100-200 | LLM API (gpt-4.1 做 Judge) |
| **Promptfoo** | 回归测试 | $30-60 | LLM API (gpt-4.1-mini 做 grader) |
| **合计** | | **$130-290/月** | |

> 参考: FinSpark 每月 12-Agent 分析的 LLM 成本本身约 $500-1000，评估体系成本约占 15-30%。

## 六、产品经理的决策框架

### 6.1 "我应该看哪个仪表板？"

| 你想回答的问题 | 去看 | 频率 |
|--------------|------|------|
| "系统还活着吗？有没有报错？" | **Langfuse** Trace 列表 | 每天 |
| "最近的分析质量怎么样？" | **OpenEvals** 趋势图 (/api/eval/trend) | 每周 |
| "新版本能不能上线？" | **Promptfoo** CI 结果 (GitHub Actions) | 每次发版 |
| "哪个 Agent 最弱？" | **OpenEvals** 维度统计 (/api/eval/stats) | 每周 |
| "这个月花了多少钱？" | **Langfuse** 成本追踪 | 每月 |
| "改了配置效果如何？" | **Promptfoo** 对比实验 (npx promptfoo view) | 按需 |

### 6.2 "出了问题找谁？"

| 问题类型 | 先看 | 再看 | 行动 |
|---------|------|------|------|
| "用户说分析失败了" | **Langfuse** (找到那次 Trace, 看哪里报错) | — | 修 bug |
| "用户说分析不准" | **OpenEvals** (查那次报告的评分) | **Langfuse** (看 Agent 的输入输出) | 调 Prompt |
| "新版本上线后质量下降" | **OpenEvals** (看趋势图拐点) | **Promptfoo** (跑一次完整评测对比) | 回滚 or 修复 |
| "开发说改了检索逻辑" | **Promptfoo** (CI 结果) | — | 通过→合并, 失败→改代码 |

## 七、总结: 三个工具的本质区别

| | Langfuse | OpenEvals | Promptfoo |
|---|---------|-----------|-----------|
| **本质** | 监控系统 | 质检系统 | 测试系统 |
| **阶段** | 生产运行时 | 生产运行时 | 开发/CI 阶段 |
| **频率** | 每次调用都记录 | 10% 抽样评估 | 每次代码变更 |
| **对象** | 系统行为 (延迟/成本/错误) | 输出质量 (准确/深度/幻觉) | 版本差异 (新 vs 旧) |
| **产出** | Trace 仪表板 | 质量评分 + 告警 | Pass/Fail + 矩阵对比 |
| **决策** | "哪里出了问题" | "质量是否达标" | "能不能上线" |
| **替代成本** | 自建需 ~20 天 | 自建需 ~15 天 | 无替代 (CI 新增能力) |
| **如果没有** | 出错后瞎猜 | 质量下滑发现不了 | 每次上线都在赌运气 |

**最终一句话**:
> Langfuse 让你**看得见**, OpenEvals 让你**判得准**, Promptfoo 让你**改得稳**。三者缺一不可,但解决的是完全不同的问题。
